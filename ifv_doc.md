# Information Fidelity Verification (IFV)
## 一种面向 LLM 书籍改写的无模型信息保真验证系统（架构设计说明书 v2.0）

---

# 1. 问题定义

## 1.1 问题背景
本系统针对大语言模型（**LLM**）将书面书籍改写为口语化播客脚本的特定场景。在此场景中，LLM 的核心任务是在**不进行信息压缩、摘要或删减**的前提下，纯粹进行表达方式的口语化重组。由于处理的文本量通常达到**数万字**，高频的上下文交互极易引发**LLM 幻觉**（Hallucination）。

传统基于深度学习的语义向量相似度（如 **Cosine Similarity**）存在“黑盒不可解释”与“对微观信息遗漏不敏感”的缺陷。因此，本系统构建了一套基于经典字符串与信息检索算法的**信息保真验证系统**（**Information Fidelity Verification**，简称 **IFV**），用于精准量化改写过程中的信息鲁棒性。

## 1.2 检测目标分类
系统需要硬性识别并定位以下**4 种核心局部变动行为**：
- **信息遗漏**（**Missing**）：原书中存在的关键事实或论据，在播客文稿中丢失。
- **信息幻觉**（**Novel**）：播客文稿中引入了原书并未提及的外部推论或事实。
- **重复表达**（**Duplicate**）：同一段原书内容被 LLM 在播客中多次换壳复述。
- **覆盖完备度**（**Coverage**）：全局视角下，原书关键信息被完整保留的结构化比例。

---

# 2. 系统设计目标与约束

本系统严格遵循**无模型**（**Model-Free**）的设计范式，具体约束如下：
- **零黑盒依赖**：严禁依赖任何机器学习、深度学习模型或外部 LLM API。
- **完全可审计**：匹配结果必须具备**确定性**与**可复现性**，能精准追溯到字符级偏移量。
- **低资源消耗**：纯 Python 环境下可高效运行，核心检索复杂度需压降至**线性时间复杂度**。

---

# 3. 总体架构流程

系统的核心逻辑是通过局部特征哈希指纹的提取与倒排索引，将全文暴力比对降维为候选局部的精确对齐。整体流水线（**Pipeline**）包含以下**10 个核心阶段**：


```

Book / Podcast Raw Text
↓
[4. Sentence Split] ——→ 建立单句级别粗颗粒边界
↓
[5. Tokenization & Lexical Normalization] ——→ 基于《同义词词林扩展版》消除口语化词汇变动
↓
[6. Shingle Pipeline] ——→ 构建局部连续词元滑动窗口
↓
[7. Rabin-Karp Rolling Hash] ——→ 将字符串窗口映射为 uint64 整型
↓
[8. Winnowing Filter] ——→ 局部最小哈希采样，实现高倍率指纹压缩
↓
[9. Fingerprint Reverse Index] ——→ 针对原书建立全局倒排索引
↓
[10. Candidate Retrieval & Dynamic Window] ——→ 召回候选集并合并相邻句，解决 N-to-M 映射
↓
[11. Fine Alignment] ——→ 利用序列对齐算法计算精确得分与差异行为
↓
[12. Coverage Analysis & Mapping] ——→ 全局状态矩阵分析
↓
Information Fidelity Report ——→ 产出包含归因列表的审计报告

```

# 4. Sentence Split（句子切分）

由于段落（Paragraph）在口语化改写中极易被 LLM 强行拆分或合并，系统统一采用**自然句**（**Sentence**）作为最小计算与映射单元。

### 4.1 切分终止符定义
系统采用以下**6 类符号**及闭合标点作为硬性切分边界：
1. **句号**（`。`）
2. **问号**（`？`）
3. **叹号**（`！`）
4. **分号**（`；`）
5. **省略号**（`……`）
6. **右引号状态**（如 `”`、`’`）与上述标点并存时的物理重合点。

切分后的输出表示为独立文本显式序列：
- 原书句集：$\mathbf{S_{Book}} = \{B_1, B_2, \dots, B_n\}$
- 播客句集：$\mathbf{S_{Podcast}} = \{P_1, P_2, \dots, P_m\}$

---

# 5. Tokenization & Lexical Normalization（分词与词法归一化）

本阶段的目标是消除书面语转口语时，因**同义词替换**（如“匮乏”转为“不够”）导致的特征指纹大规模失效（特征雪崩）。

### 5.1 分词约束
系统首选 **jieba** 等确定性分词工具进行分词。在分词完成后，实施严格的**词性过滤**（**POS Filtering**）：
- **保留词性**：名词（`n`）、动词（`v`）、数字（`m`）、英文外来词（`eng`）、专有名词（`nr`/`ns`/`nt`）。
- **过滤停止词**：结构助词（“的”、“地”、“得”）、语气助词（“了”、“着”、“过”、“吗”、“啊”）等。

### 5.2 基于《同义词词林扩展版》的归一化映射
为了在无模型状态下对语义进行对齐，系统引入哈尔滨工业大学社会计算与信息检索研究中心发布的**《同义词词林扩展版》**（哈尔滨工业大学社会计算与信息检索研究中心，2005）。该词典采用 **5 级树状层级分类**，对每个中文词汇赋予一个固定长度的**语义编码**（例如：`Da02B01=`）。

工程落地时，系统解析词典并遵循以下**3 类尾缀符号逻辑**构建静态映射内存字典（`dict`）：
- **等号尾缀（`=`）**：代表词群内各词为**绝对同义词**。系统将其内部的所有词汇（如：“人类”、“人口”、“个人”）统一硬编码强制映射为该词群的**首个原子词**。
- **井号尾缀（`#`）**：代表词群内词汇为**同类相关词**，词义存在微观语境差异。在播客口语化查重场景中，系统将其视作泛化同义词，同样进行**强行归一化降维**。
- **独立符号（`@`）**：代表该词为**无同义词独立词**，系统直接保持原词输出。

```python
# 词法归一化映射示例
# 原始 Token 流: ["创新", "推动", "资本主义", "的", "持续", "发展"]
# 过滤停止词后: ["创新", "推动", "资本主义", "持续", "发展"]
# 词林映射后:   ["创新", "推动", "资本主义", "不断", "发展"]  # "持续"与"不断"同组，完成归一化

```

# 6. Shingle Pipeline（滑动窗口构建）

**Shingle** 机制通过在过滤并归一化后的 Token 序列上建立固定长度的滑动窗口，将长文本切分为局部连续词元片段（Schleimer, Wilkerson, & Aiken, 2003）。

### 6.1 数学定义

设归一化后的句子 Token 序列为 $\mathbf{T} = [t_1, t_2, \dots, t_x]$，设定**窗口大小**（**Shingle Size**）为 $k$。则该句子生成的 Shingle 集合 $\mathbf{H}$ 满足：


$$\mathbf{H} = \{ [t_i, t_{i+1}, \dots, t_{i+k-1}] \;\vert\; 1 \le i \le x - k + 1 \}$$

### 6.2 窗口大小 $k$ 的工程权衡

为了平衡对口语化语序调整的鲁棒性以及误匹配率，不同粒度的对比维度如下表所示：

| 匹配颗粒度维度 | 空间与时间复杂度 | 对语序调整/修饰语插入的鲁棒性 | 误匹配率（False Positive） | 播客改写场景适用度 |
| --- | --- | --- | --- | --- |
| **单字/词级别 ($k=1$)** | 极低 | 极高（完全打乱依然能匹配） | 灾难级（高频常用词导致全局串扰） | **不适用** |
| **短 Shingle ($k=3 \sim 4$)** | 适中 | **优秀**（高度兼容口语化增删词汇） | 极低（结合词林归一化后表现极其稳定） | **首选推荐** |
| **长 Shingle ($k \ge 8$)** | 适中 | 极差（任意一个修饰词改变即指纹断裂） | 零误报 | **不适用** |
| **整句级别** | 极低 | 零（无法容忍任何局部的口语化修饰） | 零误报 | **不适用** |

本系统针对播客改写场景，推荐硬性初始化参数设定为：**$k = 3$**。

---

# 7. Rabin-Karp Rolling Hash（滚动哈希计算）

直接对字符串形式的 Shingle 进行比对和存储会消耗大量内存与 CPU 时钟周期。系统引入 **Rabin-Karp Rolling Hash** 算法（Karp & Rabin, 1987），将字符串视作多项式，映射为 `uint64` 类型的整形数字。

### 7.1 多项式滚动递推公式

设基数为 $B$（通常取大质数，如 $B = 31$ 或 $B = 131$），模数为 $M = 2^{64}-1$。
对于窗口移动时移出左侧词元 $t_{i-1}$ 并移入右侧词元 $t_{i+k-1}$ 的行为，新窗口的哈希值 $H_i$ 的递推计算在 $O(1)$ 时间内完成：


$$H_i = \left( (H_{i-1} - t_{i-1} \cdot B^{k-1}) \cdot B + t_{i+k-1} \right) \pmod M$$

---

# 8. Winnowing Filter（局部最小采样算法）

一本书籍在经过 Shingle 处理后会产生密集的哈希序列。为了压缩存储空间并剔除冗余信息，系统采用 **Winnowing 算法**（Schleimer, Wilkerson, & Aiken, 2003）实施局部指纹采样。

### 8.1 采样规则

设定**指纹窗口大小**（**Winnowing Window Size**）为 $w$。
在长度为 $w$ 的滑动哈希序列窗口中，系统**仅挑选数值最小的哈希值**作为该区域的代表性文本指纹（**Fingerprint**）。若存在多个相同最小值，则保留最右侧的一个。随着窗口右移，只有当原最小指纹滑出边界或遭遇更小哈希值时，才会触发新指纹的采样。

通过该算法，系统在保证漏报率为零的前提下，将指纹数据量成功压缩至原序列的 $\frac{2}{w+1}$（Schleimer, Wilkerson, & Aiken, 2003）。本系统推荐工程初始参数设定为：**$w = 4$**。

---

# 9. Fingerprint Reverse Index（指纹倒排索引）

系统在内存中针对原书（Book）的所有采样指纹构建一个高并发的**倒排索引**结构（`dict[int, list[int]]`）：


$$\mathbf{Index}: \text{Fingerprint\_Hash} \longrightarrow \{ \text{Sentence\_ID}_{\text{Book1}}, \text{Sentence\_ID}_{\text{Book2}}, \dots \}$$

该索引的查询与插入操作均可在 **$O(1)$ 恒定时间**内完成，为后续的大规模实时检索提供了算力支撑。

---

# 10. Candidate Retrieval & Dynamic Window（候选召回与动态上下文窗口）

为了应对 LLM 在生成口语化播客时高频发生的**1-to-N（单句拆多句）**和**N-to-1（多句合并）**现象，系统在召回阶段放弃了单句对单句的绝对映射限制，引入**动态上下文窗口**机制。

### 10.1 动态召回执行算法

1. **单句指纹命中统计**：遍历播客文稿中的某一特定单句 $P_j$，提取其指纹并检索倒排索引，统计各原书句子 $\mathbf{B_i}$ 的指纹命中频次（**Hit Count**）。
2. **上下文动态外扩**：当某原书单句 $B_i$ 的命中频次超过基准阈值（通常设为 $\ge 1$ 次命中）时，系统自动将该句的**物理前后相邻句**（$[B_{i-1}, B_i, B_{i+1}]$）融合成一个临时的**联合目标文本块**（**Combined Context Block**）。
3. **消除隔离碎片**：此机制确保了被 LLM 分解重组的口语化表达能够在一个连续的局部上下文中被完整召回，从根本上杜绝了因断句差异导致的误报。

---

# 11. Fine Alignment（微观精确序列对齐）

在动态窗口锁定原书候选块与播客句之后，系统进入最终的精密核验阶段。此阶段脱离哈希指纹，回归到 Token 序列本身，使用经典动态规划算法进行结构性校准。

### 11.1 算法选择

系统支持两种经典的序列对齐算法：

* **最长公共子序列**（**LCS**）：空间复杂度与时间复杂度均为 $O(L^2)$（$L$ 为句子 Token 长度）。主要通过计算两序列的最大交集，输出客观相似度得分。
* **Needleman-Wunsch 算法**：通过引入明确的打分矩阵（**Scoring Matrix**），对**匹配（Match）**、不匹配（Mismatch）**和**空隙（Gap/插入与删除）赋予不同的权重惩罚（Needleman & Wunsch, 1970）。

### 11.2 行为精准归因判定

通过对齐矩阵的回溯路径（**Backtracking Path**），系统可直接在线性时间内推导出确切的字符级变动轨迹，并完成以下功能性判定：

* **Delete 行为**：路径发生垂直偏转，判定为原书词汇被遗漏（**Missing**）。
* **Insert 行为**：路径发生水平偏转，判定为播客文稿引入了全新词汇（**Novel/幻觉候选**）。
* **Match 行为**：对角线推进，代表信息完美保留。最终单区域对齐得分公式定义为：

$$\text{Alignment\_Score} = \frac{2 \times \vert \text{LCS}(B_{\text{block}}, P_j) \vert}{\vert B_{\text{block}} \vert + \vert P_j \vert}$$



系统设定对齐核心阈值（**Alignment Threshold**）为 **$0.65$**。高于此得分的匹配关系被确认为合法映射。

# 12. Coverage Analysis（全局保真度覆盖率矩阵分析）

当所有的播客句子遍历完毕后，系统在内存中构建出一个全局的映射关联矩阵（**Mapping Matrix**），并依此计算核心质量指标：

### 12.1 覆盖率（Coverage）

$$\text{Coverage} = \frac{\text{成功建立合法映射关系的原书句子总数 } \vert B_{\text{mapped}} \vert}{\text{原书自然句物理总数 } \vert \mathbf{S_{Book}} \vert}$$

### 12.2 信息遗漏率（Missing Rate）与定位

所有在全局矩阵中**映射关系计数为 0** 的原书句子集合 $\mathbf{B_{missing}}$。此集合直接映射出 LLM 漏掉的书籍知识点。

### 12.3 幻觉率（Novel / Hallucination Rate）与定位

所有在全局矩阵中**找不到任何原书来源**的播客句子集合 $\mathbf{P_{novel}}$。此类句子是审计人员需要重点审核的“大模型自我发挥”高危区域。

### 12.4 重复复述率（Duplicate Rate）

若同一个原书单句 $B_i$ 映射到了多个不同的播客句子（即存在 $1 \to N$ 且空间跨度极大的映射点），则判定为 LLM 产生了车轱辘话的**冗余表达**。

# 13. 时间复杂度分析

假定原书包含 $N$ 个 Token，播客文稿包含 $M$ 个 Token，原书自然句平均 Token 长度为 $L$，动态窗口召回的平均候选集大小为 $K$。

各阶段的时间消耗上限分布如下：

1. **Sentence Split 与 Tokenization**：均属于单次线性扫描，复杂度为 $O(N + M)$。
2. **Shingle 与 Rolling Hash**：由于采用了滑动递推多项式算法，计算耗时为 $O(N + M)$。
3. **Winnowing Filter 采样**：单向滑窗扫描，复杂度为 $O(N + M)$。
4. **倒排索引构建**：哈希入库，复杂度为 $O(N)$。
5. **Candidate Retrieval 候选召回**：指纹匹配查询，复杂度为 $O(M)$。
6. **Fine Alignment 序列对齐**：仅在召回的局域片段内执行动态规划，总体复杂度为 $O(M \times K \times L^2)$。

### 13.1 总体性能结论

由于在实际工程中，$L$（句子长度）通常 $\le 100$ 且为常数， $K$（候选召回句数）通常 $\le 3$，因而微观对齐的计算耗时极低。系统整体的总体渐进时间复杂度近似表现为 **$O(N + M)$ 线性时间**。这意味着该系统能轻松支撑数十万字级长文本的秒级快速全量保真度验证。

# 14. 系统局限性与前沿演进

虽然本系统通过整合《同义词词林扩展版》极大缓解了近义词替换引起的误报，但由于其无模型的底层逻辑，系统依然存在以下边界局限：

* **深层句式解构无能**：若 LLM 将原书的“主动语态”彻底重构为完全不含有任何原词 token 的“被动隐喻类比表达”，系统会由于缺乏特征指纹而将其误判为 Missing。
* **事实正确性交叉校验缺失**：系统只能验证“播客文本是否包含原书的词法投影”，无法识别由于 LLM 改变修饰词（如将“高达 90%”改写为“仅有 90%”）引起的事实性逻辑颠倒。这类细微的语义级别对齐，在生产环境中仍需配合最终的人工审查机制（**Human-in-the-loop**）来兜底。

```
